## 一、资料观点 vs 本软件

| 策略 | 资料说法 | 本软件实现 | 配置项 |
|------|----------|------------|--------|
| **准点提交** | 9:00 主战场 | ✅ 默认 08:59:58 触发 | `schedule.target_time` + `advance_seconds` |
| **多波次捡漏** | 9:05/9:10/9:15 刷新捡漏 | ✅ 失败后按波次重试 | `schedule.wave_times` |
| **库存最大店** | 开源 INVENTORY | ✅ 已有 | `shop_strategy: max_inventory` |
| **避开最热店** | 网易：选偏离市中心、竞争小的店 | ✅ 新增 | `shop_strategy: min_competition` |
| **距离最近** | 开源 NEAREST | ✅ 已有 | `shop_strategy: nearest` |
| **指定门店+换店** | ken：先指定 ID，不支持则自动换 | ✅ 新增 | 账号 `shop_id` + `shop_fallback` |
| **全省选店** | 扩大门店池 | ✅ 新增 | `shop_scope: province` |
| **提前预热** | 减少 9:00 卡顿 | ✅ 提前拉门店/session | `schedule.prewarm_minutes` |

| **云手机/连点器** | 网易提及 | ❌ **不推荐**，违规风险 | — |
| **查询申购结果** | ken：18:03 查询 | ⏳ 二期 | — |
| **周末欢乐购/旅行** | ken 独立脚本 | ⏳ 二期 | — |
| **中签率翻倍** | 网易标题营销表述 | ❌ **不能保证** | 仅提高「提交质量」 |

## 二、推荐配置（综合三家思路）

### 方案 A：稳健捡漏（对齐网易 + 本仓库默认）

```yaml
shop_strategy: max_inventory
shop_scope: city
schedule:
  target_time: "09:00:00"
  advance_seconds: 2
  prewarm_minutes: 5
  wave_times: ["09:05:00", "09:10:00", "09:15:00"]
  wave_retry_failed_only: true
account_stagger_seconds: 0.5
claim_energy: true
```

### 方案 B：低竞争门店（网易「避开热门商圈」）

```yaml
shop_strategy: min_competition
shop_scope: province
```

### 方案 C：指定门店（ken「先 SHOP_ID 再 AUTO」）

在 `data/credentials.json` 对应账号增加：

```json
"shop_id": "151510100019",
"shop_strategy": "nearest"
```

并设 `shop_fallback: true`。

## 三、养号（软件只能做一部分）

资料与 [ken-iMoutai-Script](https://github.com/AkenClub/ken-iMoutai-Script) 均强调：

1. **连续每日预约** — 本工具定时即可  
2. **小茅运 / 耐力 / 旅行** — 本工具 `claim_energy` + 请 **每天在 App 签到、答题**  
3. **固定 device_id、固定城市** — 登录后勿乱改地址  
4. **中签后及时付款** — 须在 App 内操作  
5. **勿同 IP 堆几百号** — 调大 `account_stagger_seconds`，分散网络环境

## 四、重要声明

- 外部文章「成功率翻倍」指 **体验与信息准备**，不是官方中签率保证。  
- i茅台 为 **预约 + 抽签**，本软件只优化 **预约提交** 与 **选店策略**。  
- 使用 [ken-iMoutai-Script](https://github.com/AkenClub/ken-iMoutai-Script) 等开源方案请遵守其 **Apache-2.0** 许可证与平台用户协议。  
- 勿使用违规多开、云手机批量作弊；可能导致封号。
