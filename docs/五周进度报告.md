# 茅台抢单软件 — 五周开发进度报告

**报告日期**：2026-05-20

## 一、执行摘要

| 维度 | 结论 |
|------|------|
| **一期（CLI）** | 核心预约链路 **已打通**；适合个人/小规模自用与演示 |
| **合同「纯后端 + 数据库 + 查询」** | **未完整交付**；约完成合同功能量的 **35%～45%**（按可验收后台能力估算） |
| **合同「67% 抢购/中签成功率」** | **不应作为验收项**；已书面建议改为「预约提交成功率」 |
| **统一付款** | **未做**；建议改为「中签提醒 + 导出待付清单」 |
| **风险** | 登录 429、接口随 App 升级失效；需持续维护 `crypto.py` |

**总体进度（五周周期内）**：**第 1～4 周任务基本完成；第 5 周处于优化、文档与二期方案阶段。**

## 三、对照合同功能条（1）～（6）

| 合同功能 | 合同要求摘要 | 当前进度 | 完成度 | 说明 |
|----------|--------------|----------|--------|------|
| **(1) 用户与登录** | 手机号登录、信息入库、后台可查 | CLI 登录 + 本地加密 JSON | **40%** | 无数据库、无后台用户列表 API |
| **(2) 酒类管理** | 后台增删改商品参数 | `config.yaml` 手写 `items` | **15%** | 无 CRUD 界面与 API |
| **(3) 定时抢购** | 后台设时、全自动执行 | 定时 + 手动/CLI 触发 | **55%** | 缺任务队列、Cron 管理页 |
| **(4) 数据查询** | 用户、状态、抢购记录明细 | `data/run.log` 文本日志 | **20%** | 无结构化记录与查询 |
| **(5) 统一付款** | 后台集中付款、订单汇总 | 未实现 | **0%** | 建议改为提醒/导出，不做代扣 |
| **(6) 门店管理** | 绑店、门店列表管理 | 自动选店策略 | **35%** | 无门店绑定配置页、无缓存管理 API |
| **KPI：67% 成功率** | 抢购成功比例 | 未承诺 | **N/A** | 建议改为预约提交成功率 ≥67% 或 ≥90% |

**合同后台整体完成度（加权估算）**：约 **30%～40%**  
**CLI 一期完成度**：约 **85%～90%**（面向自用场景）

## 六、验收建议

### 6.1 当前阶段可验收（CLI 一期）

- CLI 菜单功能完整，可完成：配置 → 登录 → 试跑 → 定时预约 → 日志  
- 在 **≥10 个有效测试账号**、申购时段内，**预约提交成功率 ≥85%**（排除明显封号、空验证码）  
- **不以中签率、抢到比例、付款成功率为验收条件**

### 6.2 整份合同验收（需二期）

- 完成后端 API + Web 管理端 + 数据库  
- 预约提交成功率指标：**≥67%**（100 有效账号 × 3 轮）或双方另议 **≥90%**  
- 删除或变更「67% 抢到酒」「统一付款」原文
