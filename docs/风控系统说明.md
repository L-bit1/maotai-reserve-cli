# 风控系统说明

本 CLI 的风控目标：**降低多账号被平台关联、限流（429）的概率**，不保证中签。

## 架构

| 模块 | 文件 | 作用 |
|------|------|------|
| 设备画像 | `src/risk_control.py` + `device_util.py` | 每账号稳定 UA / MT-Info / 网络类型 |
| 请求节流 | `risk_control.RequestThrottle` | 发码间隔、预约频率、429 冷却 |
| 行为模拟 | `api.warmup()` | 预约前 H5 + 用户 + 场次浏览 |
| 时间错峰 | `runner` + `schedule` | 多账号 jitter、出口组间隔 |
| IP 隔离 | `proxy_pools` + `egress_group` | 见 [IP隔离部署指南.md](./IP隔离部署指南.md) |

## 推荐配置（多账号）

```yaml
antidetect:
  enabled: true
  stable_fingerprint: true   # 必开
  random_mt_info: false      # 勿每会话随机 MT-Info
  login_vcode_interval: 90
  max_reserve_per_minute: 6
  reserve_429_cooldown: 90

proxy_pools:
  ip001: "http://..."
account_stagger_seconds: 0.5
egress_group_stagger_seconds: 3
```

每个账号 `credentials` 中配置 `egress_group`，登录一次后会写入 `device_ua` / `device_mt_info`（与 `device_id` 绑定，勿手改）。

## 三层防护

1. **指纹层**：`stable_fingerprint` 按 `device_id` 固定设备特征；账号之间不同、单账号长期不变。  
2. **行为层**：预热浏览、请求间随机延迟、小茅运概率领取、预约时间 jitter。  
3. **限流层**：发码 90s 间隔；预约 6 次/分钟；遇 429 自动冷却 90s（登录 429 冷却更长）。

## 登录注意

- 勿连续点「发送验证码」  
- 遇 429 等待 5～15 分钟，换 4G 热点  
- 重新登录会保留原 `device_id` 与设备指纹  

## 自检

```bash
python check.py
```

「风控」一行应显示 `稳定指纹` 与发码/预约限额；多账号未配代理会提示 warn。

## 无法做到的事

- 不能绕过平台抽签规则  
- 不能消除同 Wi‑Fi 上千号不风控（必须代理分组）  
- 不能保证 429 永不出现（取决于平台策略与操作频率）
